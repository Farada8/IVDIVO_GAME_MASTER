## P17 — Архитектура метавселенной

### 1. Резюме

Метавселенная проекта состоит из шести слоёв, растущих концентрически вокруг базовой настольной партии:

1. **Физическая игра** (Reality Fracture / Portal Game / War of Archetypes) — ядро.
2. **Лор и кампания** — сюжетный каркас, связывающий партии в общую историю.
3. **Цифровой профиль** — учёт прогресса, коллекции и статистики игрока между партиями.
4. **Кампанейское приложение** — цифровой компаньон для ведения кампании (не замена настольной игры).
5. **Платформа сообщества** — клубы, турниры, генерируемый пользователями контент.
6. **Франшиза** — лицензирование, региональные партнёры, образовательные и психологические форматы.

### 2. Источники и версии

[APPROVED] PROJECT_CORE_CONTEXT.md v1.1 (эмоциональная дуга, формула проекта). [APPROVED] DEC-004 (синтез трёх продуктовых линий).

### 3. Каноническое ядро

Эмоциональная дуга проекта — ПЕРСОНАЖ → УБЕЖИЩЕ → ТЕРРИТОРИЯ → ФИЗИЧЕСКИЙ МИР → СВЯЗАННЫЕ МИРЫ → ПЛАНЕТАРНЫЙ КОНФЛИКТ → СИСТЕМА МИРОВ → ПРЕОБРАЗОВАНИЕ РЕАЛЬНОСТИ — задаёт порядок раскрытия слоёв метавселенной: сначала физическая игра, остальные слои открываются по мере роста масштаба кампании, а не одновременно.

### 4. Сравнения

| Референс | Что взято | Что не взято |
|---|---|---|
| Gloomhaven (цифровой компаньон) | Ведение кампании без ручного учёта | Привязка приложения к одному сценарию печати |
| Marvel Champions (модульность фракций) | Асимметричные роли как отдельные модули | Строго кооперативный формат (у нас гибрид) |
| Twilight Imperium (масштаб метавселенной) | Постепенное открытие новых систем/миров | Фиксированная карта галактики (у нас — открываемая) |

### 5. Варианты

- **Вариант A — линейное открытие**: слои метавселенной открываются строго последовательно по мере прохождения кампании. Плюс: простота для MVP. Минус: цифровой профиль не работает до поздней стадии.
- **Вариант B — параллельное открытие**: физическая игра и минимальный цифровой профиль (учёт коллекции) доступны с первой партии; лор и кампания открываются постепенно. Плюс: цифровой слой сразу даёт retention. Минус: больше работы на MVP.
- **Вариант C (рекомендуемый) — гибрид**: физическая игра + минимальный профиль сразу; кампания, платформа сообщества и франшиза — поэтапно, привязаны к контрольным точкам кампании (см. P24).

### 6. Рекомендуемая архитектура

Вариант C. Обоснование: соответствует DEC-005 (базовая коробка должна быть законченной игрой) — цифровой слой не компенсирует незаконченное ядро, а расширяет уже работающий продукт.

### 7. Спецификация

MVP цифрового профиля: ID игрока, список сыгранных сценариев, открытые миры, коллекция карт/дополнений. Хранится локально + опциональная синхронизация (не блокирует офлайн-игру).

### 8. Пример

Игрок завершает базовую кампанию (Физический мир + Убежище) → цифровой профиль фиксирует "открыт портал в Астральный мир" → при следующей покупке дополнения приложение автоматически подгружает контент для нового мира.

### 9. Риски

- Приложение может восприниматься как обязательное — необходимо явно тестировать полную играбельность без него.
- Синхронизация профиля создаёт зависимость от инфраструктуры на этапе, когда компания ещё не имеет доказанного продукта.

### 10. Прототип и тестирование

Для бумажного прототипа цифровой слой не нужен — эмулируется бумажным campaign log (см. P24). Приложение — не раньше Этапа 6 роадмапа (Kickstarter).

### 11. TO-VERIFY

Точная модель монетизации цифрового слоя (разовая покупка / free-to-play с косметикой / подписка на контент кампаний) — решение не принято.

### 12. Запись в журнал решений

Рекомендация к DEC: "Цифровой профиль — опциональное расширение, никогда не блокирующее офлайн-партию" — предложить основателю как явное решение (пока не зафиксировано).

## P18 — Иерархия миров

### 1. Резюме

Восемь миров образуют вертикальную (не параллельную) иерархию — "выше и прежде", согласно DEC-003 и принципу проекта о вертикальной архитектуре. Каждый мир имеет функцию, механическое влияние и коммерческое рабочее название.

### 2. Источники и версии

[APPROVED] PROJECT_CORE_CONTEXT.md v1.1, раздел "Восемь миров". [APPROVED] DEC-003.

### 3. Каноническое ядро

Восемь миров (рабочие названия, из PROJECT_CORE_CONTEXT.md):

| № | Мир | Игровая функция (предлагается) |
|---|---|---|
| 1 | Инфернальный / адский мир | Источник угрозы: чудовища, порча, разрушение форм |
| 2 | Физический мир | Территория, ресурсы, тактика — база кампании |
| 3 | Астральный мир | Эмоции, иллюзии, скрытая информация, блеф |
| 4 | Ментальный мир | Стратегии, изменение правил, карты-модификаторы |
| 5 | Каузальный мир | Отложенные последствия, ветвление кампании |
| 6 | Мир Души / Синтеза | Кооперация, восстановление, командные механики |
| 7 | Мир Воли / Архетипов | Глобальные законы, роли высокого уровня |
| 8 | Огненный мир | Финальная трансформация, уничтожение/пересборка |

### 4. Сравнения

| Референс | Что взято |
|---|---|
| Root (асимметрия фракций) | Каждый мир даёт разный набор доступных действий |
| Spirit Island (эскалация угрозы) | Инфернальный мир как нарастающая угроза, а не разовое препятствие |
| Terraforming Mars (постепенное открытие пластов) | Миры открываются по мере прогресса, не все сразу |

### 5. Варианты

- **A — все 8 миров доступны с первой партии**: максимальная глубина сразу, но перегружает новичка (нарушает "партия 5–10 минут объяснения" из ROADMAP Этапа 2).
- **B — базовая игра = 2 мира** (Физический + Инфернальный), остальные 6 — дополнения: соответствует установленному принципу "прототип: два плана и один вихрь" (см. предыдущие сессии).
- **C (рекомендуемый)** — как B, но с явной картой прогрессии: каждое дополнение открывает +1 мир в фиксированном порядке (Физический → Инфернальный → Астральный → Ментальный → Каузальный → Душа → Воля → Огонь), совпадающем с эмоциональной дугой проекта.

### 6. Рекомендуемая архитектура

Вариант C. Прямо реализует принцип "начать с двух планов и одного вихря" и эмоциональную дугу PROJECT_CORE_CONTEXT.md без противоречий.

### 7. Спецификация (черновая)

Каждый мир — модульная колода из ~15–20 карт + 1 планшет мира с уникальным правилом. Открытие мира = добавление его колоды и правила без изменения базового цикла хода (см. P41).

### 8. Пример

Базовая коробка: Физический мир (ресурсы, здания) + Инфернальный мир (колода угроз). Первое дополнение "Astral Rift" добавляет Астральный мир: карты иллюзий, механику скрытой информации.

### 9. Риски

- Линейный порядок открытия миров может ощущаться как "покупай по порядку" — стоит предусмотреть альтернативные входные точки для коллекционеров.
- 8 полноценных миров = 8 дополнений — большой объём производства; нужно проверить экономику на Kickstarter (см. P61) до финализации.

### 10. Прототип и тестирование

Для первого бумажного прототипа — только Физический + Инфернальный (согласуется с принципом "два плана и один вихрь"). Остальные 6 миров тестируются последовательно после подтверждения ядра.

### 11. TO-VERIFY

Коммерческие названия миров не утверждены (сейчас рабочие/описательные). Требуется отдельный проход по неймингу (связано с P07 — словарь терминологии, Блок 1).

### 12. Запись в журнал решений

Рекомендация: зафиксировать порядок открытия миров (Физический → Инфернальный → Астральный → Ментальный → Каузальный → Душа → Воля → Огонь) как рабочую гипотезу DRAFT, не APPROVED.

## P19 — Влияние высших миров на нижние

### 1. Резюме

Механика "нисходящего импульса": события/эффекты миров более высокого уровня иерархии (см. P18) могут временно менять правила, доступ или условия в мирах более низкого уровня.

### 3. Каноническое ядро

Соответствует принципу проекта: "развитие как осознанное возвращение дифференцированного к единому Источнику" — высшие миры не просто сильнее, а меняют сами правила нижних.

### 5. Варианты

- **A — импульс как карта события**: карта из колоды высшего мира разыгрывается и модифицирует текущий раунд в нижнем мире (простое, легко тестировать).
- **B — импульс как постоянный статус-эффект**: открытие высшего мира даёт постоянный модификатор всем нижним мирам до конца партии (даёт ощущение веса, но рискует power creep).
- **C (рекомендуемый) — гибрид с ограниченным ресурсом**: импульсы — это карты события с ограниченным количеством использований за партию (2–3), стоимость = Накал (см. существующий accepted-паттерн "Накал overflow penalty" из механической библиотеки проекта).

### 6. Рекомендуемая архитектура

Вариант C — переиспользует уже принятый паттерн Накала, не вводит новую валюту.

### 9. Риски

Прямое взаимодействие "выше меняет правила ниже" может ощущаться как runaway leader, если не ограничено (проект уже отклонял Feedback Snowball как паттерн — см. историю решений по механикам). Обязательно вводить контрмеру.

### 11. TO-VERIFY

Точный список того, какие параметры нижних миров может менять импульс (только правила раунда или также экономику ресурсов) — не решено.

## P20 — Обратное восхождение

### 1. Резюме

Зеркальная механика к P19: действия в нижних мирах накапливают "отклик", открывающий доступ, эффекты или преобразование в высших мирах — реализует "восходящую" половину каузальной петли.

### 5. Варианты

- **A — накопительный счётчик** (Отклик): любое значимое действие в нижнем мире добавляет 1 очко Отклика; при достижении порога открывается доступ к действию в высшем мире.
- **B — конкретные триггеры**: только определённые действия (не любые) генерируют Отклик — требует явного списка триггеров на старте.
- **C (рекомендуемый)** — B, с явным списком из 3–5 триггеров на мир, расширяемым в дополнениях.

### 9. Риски

Без верхнего предела Отклик превращается в снежный ком (нарушение принципа против Feedback Snowball). Обязательно: убывающая отдача (diminishing returns) или TTL на накопленный Отклик — переиспользовать уже принятый паттерн TTL decay.

### 12. Запись в журнал решений

Рекомендация: явно связать P19 (импульс сверху) и P20 (отклик снизу) единой формулой каузальности — оформить в P22 (карта причинности) как единый граф, не два отдельных механизма.

## P21 — Механика порталов

### 1. Резюме

Портал — не телепорт, а объект с полным набором параметров, обязательным по PROJECT_CORE_CONTEXT.md: источник, назначение, условия открытия, ключ/допуск, стоимость, риск, пропускная способность, направление воздействия, последствия переноса, уязвимость к контролю/повреждению/заражению/блокировке.

### 3. Каноническое ядро

[APPROVED, дословно из PROJECT_CORE_CONTEXT.md] "Портал не является простым перемещением. Каждый портал должен иметь источник, назначение, условия открытия, ключ или допуск, стоимость, риск, пропускную способность, направление воздействия, последствия переноса и возможность контроля, повреждения, заражения и блокировки."

### 7. Спецификация (черновая карточка портала)

| Параметр | Тип | Пример значения |
|---|---|---|
| Источник / Назначение | 2 мира из списка P18 | Физический → Астральный |
| Условия открытия | Действие + ресурс | 2 Действия + 1 Ключ |
| Пропускная способность | Число сущностей за ход | 1 персонаж / ход |
| Стоимость | Ресурс | 2 Накала |
| Риск | Таблица события | d6: 1–2 заражение, 3–5 успех, 6 критический успех |
| Уязвимость | Флаги состояния | Может быть заблокирован картой "Печать" |

### 8. Пример

Портал "Разлом Убежища": Физический → Инфернальный, стоимость 2 Накала, пропускная способность 1/ход, риск заражения 33%. Соперник может сыграть карту "Печать", блокируя портал на 1 раунд.

### 9. Риски

Слишком много параметров на старте усложнит первый прототип — для MVP допустимо урезать до 4 обязательных параметров (источник/назначение, стоимость, риск, пропускная способность), остальные добавлять в дополнениях.

### 10. Прототип и тестирование

Для бумажного прототипа — версия с 4 параметрами (см. риски). Полная версия — после первого раунда плейтестов.

## P22 — Карта причинности

### 1. Резюме

Формализация порядка разрешения эффектов между мирами: приоритеты, защита от циклов (P19 → P20 → P19 → ... бесконечно).

### 6. Рекомендуемая архитектура

Единый causal graph: Импульс (P19) и Отклик (P20) разрешаются в фиксированной фазе раунда (после основных действий, до конца хода), в порядке "сверху вниз, затем снизу вверх" за один проход — без повторных итераций в том же раунде. Это исключает бесконечные циклы by design, не через дополнительные правила проверки.

### 9. Риски

Если разрешать эффекты в реальном времени по мере триггеров (а не фиксированной фазой), возможны бесконечные циклы и неоднозначность порядка между игроками. Фиксированная фаза — обязательное решение, не опция.

### 11. TO-VERIFY

Нужно ли более одной фазы разрешения за раунд в кампаниях позднего этапа (много миров одновременно) — требует теста, не решено на бумаге.

## P23 — Состояния миров

### 1. Резюме

Каждый мир проходит через состояния: зарождение → развитие → кризис → преобразование → стабилизация → переход. Состояние определяет доступные действия и модификаторы.

### 7. Спецификация (черновая state machine)

```
ЗАРОЖДЕНИЕ → РАЗВИТИЕ → КРИЗИС → (ПРЕОБРАЗОВАНИЕ | СТАБИЛИЗАЦИЯ) → ПЕРЕХОД → (в следующий цикл ЗАРОЖДЕНИЯ или в СЛЕДУЮЩИЙ МИР по P18)
```

Триггеры перехода между состояниями = накопление конкретных ресурсов/событий мира (уточняется по каждому миру отдельно в дополнениях).

### 8. Пример

Физический мир в состоянии КРИЗИС: удвоенная стоимость строительства, но открыт доступ к аварийным действиям недоступным в обычном состоянии РАЗВИТИЕ.

### 9. Риски

8 миров × 6 состояний = потенциально 48 уникальных наборов модификаторов — избыточная сложность для MVP. Для базовой коробки ограничить 3 состояниями (Развитие / Кризис / Стабилизация), расширять в дополнениях.

## P24 — Метагалактическая кампания

### 1. Резюме

Кампания как последовательность 8–16 сценариев (согласуется с P50, Блок 7), в которой открытие миров (P18), портальная механика (P21) и каузальная петля (P19/P20) объединяются в связный прогресс с ветвлениями.

### 5. Варианты

- **A — линейная кампания**: фиксированная последовательность сценариев, без ветвлений. Проще для MVP.
- **B — ветвящаяся кампания**: выбор в конце сценария меняет следующий сценарий (legacy-модель). Выше реиграбельность, выше сложность производства.
- **C (рекомендуемый для MVP) — линейный костяк + локальные развилки**: основная последовательность фиксирована (соответствует DEC-006, "первым результатом является Game Core и минимальный прототип"), но 2–3 точки в кампании дают развилку из 2 вариантов — управляемый объём тестирования при сохранении ощущения выбора.

### 6. Рекомендуемая архитектура

Вариант C для MVP; полноценный legacy/ветвящийся формат (Вариант B) — кандидат для флагманского дополнения War of Archetypes после подтверждения ядра.

### 10. Прототип и тестирование

Минимальный бумажный прототип кампании — 3 сценария (не 8–16): старт, одна развилка, финал. Полная кампания собирается только после подтверждения core loop.

### 12. Запись в журнал решений

Рекомендация: зафиксировать "линейный костяк + 2–3 локальные развилки" как рабочую гипотезу для MVP-кампании (DRAFT, требует утверждения основателем).
